# 改进领域对象并接入 Svelte

## 一、作业背景

在 Homework 1 中，你已经围绕 `Sudoku` / `Game` 完成了一轮领域对象设计，并实现了 Undo / Redo。

但在实际提交中，出现了一个普遍问题：

> 许多同学已经“设计出了领域对象”，却没有将它真正接入现有 Svelte 游戏流程中。

结果是：

- 领域对象存在于独立模块中，但 UI 没有真正使用它
- 组件仍然直接操作旧状态或旧逻辑
- `Sudoku` / `Game` 只在测试中可用，在真实界面中没有成为核心

本次 Homework 1.1 的目标，就是解决这个问题。

---

## 二、作业目标

请在老师课堂讲解的基础上，完成以下两项任务：

### 任务 1：改进 Homework 1 中的领域对象

你需要根据课堂讲解与自己的反思，改进已有的 `Sudoku` / `Game` 设计。

### 任务 2：将领域对象接入真实 Svelte 游戏流程

你必须让现有 Svelte 前端真正消费你的领域对象，而不是只保留“独立可测试但未接入”的实现。

---

## 三、本次作业的核心问题

本次作业重点回答两个问题：

1. **Svelte 的响应式机制如何与领域对象协作？**
2. **View 层如何消费 `Sudoku` / `Game`？**

你必须在代码与文档中清楚回答这两个问题。

---

## 四、技术背景与边界

本 starter repo 当前使用的是 **Svelte 3** 风格工程，而不是 Svelte 5。  
因此，本次作业应基于以下机制工作：

- store / custom store
- 组件中的 `$store`
- 顶层 `let`
- 顶层 `$:` reactive statements

本次作业**不要求**迁移到 Svelte 5，不要求使用 runes，不要求使用 reactive classes。

## B. 将领域对象真正接入 Svelte 真实使用流程

这是本次作业最重要的要求。

你必须让真实游戏界面中的主要流程，真正通过你的领域对象完成。

### 至少需要接入以下流程：

1. **开始一局游戏**
    - 创建 `Game`
    - 创建或加载 `Sudoku`

2. **界面渲染当前局面**
    - UI 中看到的 grid 必须来自你的领域对象，或来自由你的领域对象导出的响应式视图状态

3. **用户输入**
    - 用户在界面中进行输入时，必须调用你的 `Game` / `Sudoku` 接口，而不是直接操作旧数组

4. **Undo / Redo**
    - 界面中的撤销 / 重做必须调用你的领域对象逻辑

5. **界面自动更新**
    - 当领域对象发生变化后，Svelte 界面必须正确刷新

## 六、推荐的接入方式

你可以自行设计，但必须属于“View 真正消费领域对象”的方案，而不是“领域对象存在但 UI 没用”。

下面给出两种推荐方案。

### 方案 A：Store Adapter（推荐）

建立一个面向 Svelte 的适配层，例如：

- `createGameStore(...)`
- `createSudokuStore(...)`

该适配层负责：

- 持有你的 `Game` / `Sudoku`
- 对外暴露可被 Svelte 消费的响应式状态
- 对外暴露 UI 可调用的方法

例如：

- 响应式状态：`grid`, `invalidCells`, `won`, `selectedCell`
- 响应式命令：`guess`, `undo`, `redo`

这是本次作业最推荐的方式。

---

### 方案 B：领域对象自身实现可订阅接口

你也可以让 `Game` 或 `Sudoku` 本身直接具备可订阅能力，并被组件直接消费。

但你必须自己处理好：

- 更新通知
- 订阅与取消订阅
- Svelte 的消费方式

如果你选择这种方案，必须在 `DESIGN.md` 中解释清楚其工作原理。

---

## 七、明确要求：不能这样做

以下做法**不满足**本次作业要求：

### 1. 领域对象只存在于测试中

例如：

- 测试里用了 `Sudoku`
- 但真实界面仍然走原先旧逻辑

这种情况视为“未接入真实使用”。

### 2. 只把旧数组包了一层名字

例如：

- UI 仍然直接改二维数组
- 只是顺手把它叫作 `sudoku`

这种情况不算完成领域对象接入。

### 3. 关键逻辑继续放在组件中

例如：

- Undo / Redo 仍主要写在 `.svelte` 文件里
- `guess` 逻辑散落在组件事件处理函数中

这种情况不符合本次作业要求。

## 八、关于 Svelte 响应式机制，你必须理解的事情

本次作业要求你真正理解，而不是只靠 AI “拼通”。

你至少需要理解以下问题：

1. 为什么修改对象内部字段后，界面不一定自动更新？

3. 为什么 store 可以被 `$store` 消费？

你不一定要在作业中完整讲透所有细节，但你必须在 `DESIGN.md` 中清楚解释：

- 你的方案依赖了 Svelte 的什么机制
- 你的 UI 为什么会更新
- 如果错误地直接 mutate 对象，会出什么问题
